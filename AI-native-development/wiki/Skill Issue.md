# Skill Issue

[[Andrej Karpathy]]'s framing for the default attribution pattern in the agent era: when something doesn't work, the assumption is that **you** haven't found the right way to use what's available — not that the capability isn't there.

> "Even if they don't work, to a large extent you feel like it's skill issue. It's not that the capability is not there. It's that you just haven't found a way to string together what's available."

## Examples

- "I didn't give good enough instructions in the AGENTS.md file"
- "I don't have a nice enough memory tool that I put in there"
- "I just haven't found how to parallelize them correctly"

## Why It's Empowering

> "It's a skill issue, which is very empowering, because you could be getting better. So that's why I think it's very addictive, because there's unlocks when you get better."

The skill issue framing turns failure into a learning opportunity rather than a wall. It creates a feedback loop: try → fail → attribute to skill → improve → unlock → feel rewarded → try harder.

## Why It's Destabilizing

The same framing feeds [[AI Psychosis]]: if everything is a skill issue, then there's always more you could be doing. The possibility space is infinite, and your limitations are your own fault. There's no "the tool can't do this" escape hatch.

## The Counter-Caveat

Karpathy also acknowledges [[Jaggedness]] — some things genuinely don't work yet because models are "off the rails" in non-verifiable domains. So the skill issue framing isn't always correct: sometimes it really is a capability limitation. "So it's hard to tell."
